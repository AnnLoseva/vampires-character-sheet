# i18n (RU / EN)

## Purpose
The app is bilingual (Russian / English). The UI can switch languages, and VTM
rules content exists in both RU and EN. The central risk is **identity by
translated name** — comparing display names instead of stable IDs.

## Main files
- React side:
  - `lib/i18n/LanguageProvider.tsx` — context provider; exposes `useLang()` and
    the `t()` translator used across React components.
  - `lib/i18n/dictionary.ts` — UI string translations.
  - `lib/i18n/ruleNames.ts` — maps rule display names ↔ stable identifiers.
- Legacy side (inside the iframe sheet):
  - `public/i18n-runtime.js` — legacy translation runtime.
  - `public/i18n-dictionary.js` — legacy UI strings.
- Rules content: `public/rules.json` (RU) and `public/rules_eng.json` (EN) — see
  `rules-data.md`.

## RU / EN names
- Traits, clans, disciplines, skills, merits, flaws, and predator types all have
  Russian and English names.
- The RU rules file is primary and much larger; EN must track it structurally.
- Display names differ by language and **must not be used as keys** for logic,
  lookups, comparisons, or persistence.

## The core risk: comparing display names instead of stable IDs
- Always match/store by **stable id** (or canonical key), then translate for
  display via `ruleNames.ts` / dictionaries.
- Bugs appear when code does `name === 'Fortitude'` / `=== 'Стойкость'` — this
  breaks the moment the language changes. Look for such comparisons when touching
  rules-driven logic.
- The two legacy i18n files and the React i18n must agree on the same underlying
  ids.

## Safe edit protocol
1. When adding a rule/trait, add RU + EN with a matching stable id in both rules
   files and both i18n layers as needed.
2. Never branch logic on a translated string; branch on the id.
3. Keep `lib/i18n/ruleNames.ts` the single mapping authority on the React side.
4. Verify by switching language and confirming rules, names, and any
   name-dependent logic still resolve.
